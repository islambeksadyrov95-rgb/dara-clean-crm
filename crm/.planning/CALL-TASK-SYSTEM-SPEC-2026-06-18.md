## 0. Цель и принцип

**Звонок никогда не заканчивается «никак».** Каждый исход = либо завершённое действие (заказ),
либо **автоматически запланированная следующая задача** с умным дефолтом времени. Очередь `/queue`
становится единым списком задач («Today»): прячет клиента до срока, возвращает его due-first.

Метрика здоровья (по Pipedrive «activity-based selling»): **доля активных клиентов с будущей задачей ≈ 100%.**
Клиент без следующей задачи и без терминального архива = утечка (сейчас таких сценариев 3–4, см. §2).

Бизнес-мотор Dara Clean — **retention/повторные продажи** (обзвон базы прошлых клиентов), НЕ воронка
новых лидов. Новый клиент входит в пул через первый заказ (Agbis-импорт/создание) и сразу подчиняется
retention-плейбуку. Отдельная acquisition-воронка не нужна.

## 1. Lifecycle-модель клиента (retention)

Клиент в каждый момент в одном из состояний. Состояние ВЫЧИСЛЯЕМОЕ (из последнего call_log + `next_action_at` +
RFM), отдельной колонки-стадии не заводим (как сейчас архив вычисляется во view).

| Состояние | Что значит | Как определяется | Виден в /queue? |
|---|---|---|---|
| **Свежий** | давно не звонили, созрел по давности | нет открытой задачи, `next_action_at` null/в прошлом | да (по пресету сегмента) |
| **С задачей (отложен)** | есть запланированное касание в будущем | `next_action_at` в будущем | нет — до срока |
| **Задача на сегодня/просрочена** | пора звонить | `next_action_at` ≤ now | да, **в приоритете (сверху)** |
| **Терминальный архив** | отказ/неверный/3 страйка | последний call_log status ∈ {declined, not_relevant} | нет |
| **Won (недавний заказ)** | только что оформил | last_order свежий + post-order re-touch запланирован | временно скрыт пресетом + задача на re-touch |

RFM-сегмент (`rfm_segment`, есть) остаётся слоем приоритизации СВЕЖИХ; `next_action_at` — слоем задач.

## 2. Текущая правда из кода (что чиним)

> Точно из `app/(protected)/queue/actions.ts`, `queue-query.ts`, миграций. См. разбор движка.

**Корень: два НЕсвязанных канала «будущего»:**
1. `clients.next_action_at` — ЕДИНСТВЕННОЕ поле, влияющее на видимость в очереди (snooze-фильтр
   `queue-query.ts:107`, due-first сортировка). Пишут только `snoozeClient` и форма карточки.
   **`recordDisposition` его НЕ пишет.**
2. `call_logs.next_call_date/time` — пишет `recordDisposition`. Surface только виджет «перезвоны» +
   бейдж, оба жёстко фильтруют `status='callback'`.

**Дыры (исправляются этой спекой):**
- **GAP-1 (P1-БАГ):** «Недоступен +4ч» пишет `next_call_date` со `status='not_reached'` → не виден
  НИГДЕ (виджет ждёт `status='callback'`), `next_action_at` не трогается. Тост «через 4 часа» врёт.
- **GAP-2 (архитектура):** два канала не связаны. «Перезвон» виден в виджете, но клиент НЕ уходит из
  основной очереди → **двойная видимость**.
- **GAP-3:** «Заблокировал» — без задачи, без архива, без метки. Возвращается как обычный.
- **GAP-4:** заказ без авто-follow-up (нет NPS/re-touch). Повторные продажи только на пассивном «старении».
- **GAP-5:** «Не сезон» архивирует НАВСЕГДА (хотя по смыслу — отложенный перезвон).
- **GAP-6:** «Недоволен качеством» молча архивирует — без эскалации руководителю.
- **GAP-7:** WhatsApp — разовый, без задачи «дождаться ответа».
- **GAP-8:** авто-3-страйка архивирует молча, UI не сообщает менеджеру.

## 3. Целевая модель

### 3.1. Единый механизм — `clients.next_action_at` как источник правды

- **`recordDisposition` становится ЕДИНСТВЕННЫМ местом, ставящим следующую задачу.** Любой нетерминальный
  исход пишет `clients.next_action_at` = срок (по дефолту §8) + `next_action_type` (зачем) + `next_action_note`.
- Терминальные исходы (отказ/неверный/3-страйка/DNC) — `next_action_at = null` + архив (как сейчас).
- Очередь `/queue` уже фильтрует по `next_action_at` — **она автоматически становится списком задач.**
- Виджет «перезвоны на сегодня» и бейдж: переключить на чтение `clients.next_action_at` (due сегодня,
  владелец = я) ВМЕСТО `call_logs.status='callback'`. Тогда нет двойной видимости и нет невидимого +4ч.
- `call_logs.next_call_date/time` остаётся в истории (аудит), но НЕ источник планирования.

**Модель «одна открытая задача на клиента»** (как Pipedrive «one next activity»): достаточно полей на
`clients`, отдельная таблица `tasks` НЕ нужна на их масштабе (один каденс на клиента). См. §6.

### 3.2. Матрица «исход → задача» (контракт реализации, ПОДТВЕРЖДЕНО)

`next_action_type` (новый text-enum): `callback | retry`. Всё остальное (сезон/нет денег/думает) — это
ПРИЧИНА на перезвоне/отказе, а не отдельный тип задачи.

| Исход (кнопка) | status / sub_status | `next_action_at` ← | Причина | Архив? | Доп. эффект |
|---|---|---|---|---|---|
| **Заказ** | reached/ordered | — | — | нет | форма заказа (post-order рассылки — ПОТОМ, вне спеки) |
| **Перезвон** | callback/callback_later | **+2д дефолт** или конкретная дата/время | **опц.** (тег) | нет | type=callback; быстрый выбор времени, без обязательного ручного ввода |
| **Отказ** | declined/`reason` | null | **ОБЯЗАТЕЛЬНА** | **да** | причина из списка (§8) → пишется структурно для фильтра |
| **В рассылку** | reached/added_broadcast *(new)* | null (рассылка = следующее касание) | опц. | нет | добавить клиента в рассылку (для «ушёл к конкуренту» / nurture) |
| **Не дозвонился** | not_reached/unavailable | **каденс +4ч→+1д→+3д** по № попытки (retry) | — | на 3-й → да | 3-strike сохраняется, срок теперь ВИДЕН (фикс GAP-1) |
| **Неверный номер** | not_relevant/wrong_number | null | — | **да** | терминал |
| **Заблокировал** | not_relevant/blocked | null | — | **да** | архив сразу (решение владельца; фикс GAP-3 — больше не висит без шага) |

«Думает / посоветуется / нет денег / у конкурента / не сезон» = **ПРИЧИНЫ на Перезвоне** (менеджер сам ставит когда,
или дефолт +2д). «Дорого / не нужно / недоволен качеством / не сезон» = **ПРИЧИНЫ на Отказе**. Один тег, не 20 кнопок.

### 3.3. Причины — структурный enum (фильтруемый)

`reason` (структурный, на call_logs + проброшен на clients для фильтра §5):
`думает · посоветуется · нет_денег · дорого · у_конкурента · не_нужно · недоволен_качеством · не_сезон · другое(+free text)`.

- На **Отказе — обязательна** (UI не даёт сохранить отказ без выбора). На **Перезвоне — опционально**.
- Маппинг на текущие sub_status: дорого=decline_expensive, у_конкурента=decline_competitor, не_нужно=decline_not_needed,
  недоволен_качеством=decline_quality, не_сезон=decline_season, другое=decline_other. НОВЫЕ как причины перезвона:
  думает, посоветуется, нет_денег. → нужен единый `reason`-столбец, общий для callback и declined (а не только decline_*).
- Жалоба/недоволен качеством — это ПРИЧИНА (недоволен_качеством), без отдельной эскалации. DNC не вводим.

### 3.4. Кнопки cockpit (full flow /queue) — итог

- **Primary (дозвонился, 3 цветных):** `Заказ` `1` · `Перезвон` `2` · `WhatsApp` `4`
  - **Перезвон**: раскрывает быстрый выбор «когда» (чипы +2ч·завтра·**+2д (дефолт)**·своя дата/время) + опц. причина-тег.
- **Secondary (видимая):** `Не дозвонился` `0` (авто-каденс)
- **Overflow «Другое ▾»:** `Отказ` `3` *(→ обязательный выбор причины)* · `В рассылку` · `Неверный номер` · `Заблокировал` · `WhatsApp (не дозв.)`
- Хоткеи: 1/2/3/4/0 (есть) + глобальные `C`=Позвонить, `W`=WhatsApp, `N`=следующий, Esc=закрыть.
- Авто-переход к следующему после любого терминального/запланированного исхода (есть).
- История звонков + VPBX-записи **свёрнуты по умолчанию** во время звонка (раскрыть кликом).
- Единые подписи через общий `STATUS_LABELS` (дедуп с `calls/page.tsx`).
- **Отказ нельзя сохранить без причины** (кнопка «Сохранить» в фазе отказа disabled, пока причина не выбрана) — на обоих flow (queue + clients).

## 6. Изменения данных

- `clients.next_action_at timestamptz` — **есть**, становится источником правды.
- **+ `clients.next_action_type text`** CHECK in (`callback`, `retry`) — зачем задача (для бейджа в строке очереди §5).
- **+ `clients.next_action_note text NULL`** — заметка к касанию (свободный текст менеджера).
- **+ structured `reason`** — единый столбец-причина, общий для callback и declined (НЕ только decline_*),
  значения §8.4. На `call_logs.reason` (есть text-колонка — сделать структурным enum) + проброс последней
  причины на `clients` (колонка `last_call_reason text NULL`) для ФИЛЬТРА в списке клиентов/очереди (§8.8).
- `call_logs.next_call_date/time` — остаётся для истории (НЕ источник планирования).
- CallSubStatus enum (`actions.ts:19`) += `added_broadcast` (кнопка «В рассылку»). Причины думает/посоветуется/
  нет_денег живут в `reason`, НЕ в sub_status. `blocked` меняет родителя на `not_relevant` (теперь архивирует §3.2).
- Миграция: одна на колонки + одна на backfill `next_action_at` из существующих открытых `callback`-логов
  (чтобы текущие перезвоны не потерялись при переключении виджета на новый канал).
- `database.ts` регенерировать (gen:types) в том же коммите.

**Решение [arch]:** одна открытая задача на клиента через поля `clients`, а не таблица `tasks`. Причина:
масштаб (один каденс/клиент), retention-мотор, нет нужды в нескольких параллельных задачах. Пересмотреть,
если появится потребность в нескольких одновременных напоминаниях на клиента.

## 9. Метрики успеха

- **Доля активных клиентов с будущей задачей ≈ 100%** (главная; сейчас дыры).
- **Dead-end rate = 0** (звонков без задачи и без архива).
- Конверсия перезвонов «сделанных в срок» (задача → звонок в день срока).
- (Существующие) звонки/заказы/выручка/WhatsApp в дневном плане.

## 10. Что НЕ входит (явно)

- Acquisition-воронка для новых лидов (Dara — retention-мотор, новый входит через заказ).
- Полноценный rules-engine (захардкоженные дефолты достаточно на масштабе).
- Wazzup inbound «ответ → наверх очереди» (нет webhook; backlog).
- Несколько параллельных задач на клиента (одна открытая — §6).
