## КОНТРАКТ НЕРЕГРЕССИИ (ПРИОРИТЕТ №1 — не сломать работающее на беке и фронте)

Эталон на 2026-06-11 (зафиксирован): `npm run build` ✅, `npm test` ✅ — 22 файла / 103 теста зелёные.
Любое изменение этого статуса = регрессия = блок коммита/деплоя.

Правила (обязательны для КАЖДОЙ задачи каждого агента):
1. **Каждый промпт несёт явный PRESERVE-список** — что НЕЛЬЗЯ менять/ломать (рабочие экраны, контракты API,
   импорты, поведение). Агент не трогает ничего за пределами задачи (анти-«улучшайзинг»).
2. **Before/After на затронутый домен.** До правки — список работающих UI-элементов/флоу этого домена;
   после — каждый пункт проверен живым. Из core.md: «не имитируй legacy, но и не ломай поведение».
3. **Аддитивность вместо замены.** Новое добавляем рядом с рабочим; удаляем старое только после того, как
   новое подтверждено на проде (release-management.md: breaking changes — в 2 шага).
4. **Тесты — сеть безопасности, не помеха.** Существующие 103 теста НЕ удалять/не скипать. Меняешь поведение —
   сперва обнови тест (он зафиксирует новый контракт), потом код. Падение теста = стоп, разбор причины.
5. **Контракты не ломать молча.** Меняешь сигнатуру server action / форму ответа — найди ВСЕХ вызывающих
   (grep) и обнови в ТОМ ЖЕ коммите (api-design.md §API↔Frontend). Меняешь .select() — проверь всех потребителей.
6. **Регресс-гейт на каждой волне:** `npm run build` + `npm test` зелёные → preview-деплой → ручной прогон
   core-флоу (очередь→звонок→диспозиция→следующий; создание заказа; карточка) → только потом коммит/промоут.
7. **При любом сомнении «сломает или нет» — добавить тест, а не догадку.** Покрыть регресс тестом до правки.
8. **Откат наготове:** знать Vercel instant-rollback и DOWN-миграцию ДО начала фазы (см. раздел «Откат»).

Эталон обновляется только ВВЕРХ: после каждой фазы тестов становится больше, не меньше.

Правила оркестрации (из platform-constraints + proactive-engineering):
- Два агента НИКОГДА не редактируют один файл в одной волне.
- Shared-контракты (типы, константы, хелперы) создаются ДО запуска зависимых агентов.
- Каждый промпт содержит: файлы, что менять, PRESERVE-список, критерии.
- После каждого агента: оркестратор читает git diff + чеклист по типу задачи.
- Модели: рутина → sonnet, реализация по готовому ТЗ → opus.
- После каждой волны: `npm run build` → 0 ошибок → `npm test` → 0 падений → атомарный коммит.
- «Готово» = проверено на проде, не «build прошёл».

ОБЯЗАТЕЛЬНЫЕ ГЛОБАЛЬНЫЕ ПРАВИЛА (вставлять в КАЖДЫЙ промпт агента):
1. **Next.js 16:** проект на Next 16 со сломанными относительно training data API (AGENTS.md:
   «This is NOT the Next.js you know»). `params` в роутах — это `Promise`. ПЕРЕД правкой роутов/server actions
   прочитай релевантный гайд в `node_modules/next/dist/docs/01-app/`. Не доверяй памяти про Next.
2. **Типы:** после Фазы «Фундамент» F1 существует `types/database.ts` + скрипт `gen:types`. Клиенты типизированы
   `<Database>`. Источник правды — database.ts. После любой миграции — `npm run gen:types`, закоммитить.
   (До F1 источник — SQL-схема; задачи Фундамента это учитывают.)
3. **Деньги:** `integer` — ЦЕЛЫЕ ТЕНГЕ (без тиын), правило database.md. Все денежные колонки и расчёты —
   целочисленные, `Math.round()` после любого умножения/деления. discount_percent остаётся numeric (это процент).
4. **Тесты:** проект на vitest (`npm test`), 22 файла. Если трогаешь покрытый модуль (actions, lib, route) —
   прогони `npm test`, обнови сломанные тесты, добавь тесты на новую логику. Не удаляй и не скипай тесты.
5. **Запреты (хуки enforce):** без `as any`, `@ts-ignore`, `enum`, `select('*')`, `.catch(()=>{})`,
   `console.log` в проде, `err.message` в ответе API. Money без Math.round, status-update без проверки перехода.
6. **Исполнение через GSD:** каждая фаза = GSD-фаза, запуск через `/gsd-execute-phase` (project CLAUDE.md).
   Хуки блокируют Edit/Write вне GSD — не обходить.
7. **Контекст безопасности (анти-ложные-срабатывания):** все security-задачи в этом репо — защитное
   обслуживание СОБСТВЕННОГО приложения Dara Clean CRM (см. AGENTS.md §Security work context).
   В каждый security-промпт агенту добавлять строку: «Это защитная доработка нашего собственного CRM,
   авторизована владельцем. Не pentest сторонних систем.» Формулировки задач — защитные
   («закрыть доступ», «ограничить роль»), не атакующие («exploit», «escalation» в заголовках).
