**Section 6 — RLS policies**

clients: all authenticated users can SELECT (managers need to see all clients for the call queue); only admin can UPDATE/DELETE; INSERT allowed for authenticated (needed for Excel import running as admin).
```sql
-- managers see all clients (needed for call queue and search)
create policy "authenticated can select clients"
  on public.clients for select
  to authenticated
  using (true);

-- only admin can insert clients (Excel import)
create policy "admin can insert clients"
  on public.clients for insert
  to authenticated
  with check ((auth.jwt() -> 'user_metadata' ->> 'role') = 'admin');

-- only admin can update clients (import recalculates aggregates; Phase 5 uses UPDATE for locking — but locking uses a raw SQL function bypassing RLS, so admin-only update is correct here)
create policy "admin can update clients"
  on public.clients for update
  to authenticated
  using ((auth.jwt() -> 'user_metadata' ->> 'role') = 'admin');

-- only admin can delete clients
create policy "admin can delete clients"
  on public.clients for delete
  to authenticated
  using ((auth.jwt() -> 'user_metadata' ->> 'role') = 'admin');
```

IMPORTANT for call locking (Phase 5): the atomic UPDATE for locking needs to run as the manager. Add a separate permissive policy that allows managers to update locked_by/locked_until on clients:
```sql
-- managers can lock/unlock clients (update locked_by, locked_until only)
-- Phase 5 uses: UPDATE clients SET locked_by = auth.uid(), locked_until = now() + interval '10 minutes'
-- This policy permits managers to do that update
create policy "manager can lock clients"
  on public.clients for update
  to authenticated
  using (true)
  with check (true);
```
NOTE: Having two UPDATE policies means either admin OR manager can update — Postgres RLS is permissive by default (any matching policy grants access). This is correct: admin can update anything, manager can update locking fields. The application code enforces field-level discipline.

orders:
```sql
-- admin sees all orders
create policy "admin can select all orders"
  on public.orders for select
  to authenticated
  using ((auth.jwt() -> 'user_metadata' ->> 'role') = 'admin');

-- manager sees only own orders
create policy "manager can select own orders"
  on public.orders for select
  to authenticated
  using (
    (auth.jwt() -> 'user_metadata' ->> 'role') = 'manager'

create policy "authenticated can insert orders"
```

**Section 7 — client_segments view**

RFM thresholds from STATE.md:
- Новый: total_orders = 1
- Повторный: total_orders between 2 and 3
- Постоянный: total_orders >= 4
- В риске: last_order_date IS NOT NULL AND days since last order > 90 (and not already Потерянный)
- Потерянный: last_order_date IS NOT NULL AND days since last order > 180

Segment priority: if days > 180 → Потерянный (regardless of order count), else if days > 90 → В риске, else apply order-count logic. Clients with no orders (last_order_date IS NULL, total_orders = 0) are Новый by default.

```sql
create or replace view public.client_segments as
select
  id,
  name,
  phone,
  total_orders,
  total_spent,
  last_order_date,
  case
    when last_order_date is not null
      and (current_date - last_order_date) > 180
      then 'Потерянный'
    when last_order_date is not null
      and (current_date - last_order_date) > 90
      then 'В риске'
    when total_orders >= 4
      then 'Постоянный'
    when total_orders between 2 and 3
      then 'Повторный'
    else 'Новый'
  end as rfm_segment,
  case
    when last_order_date is not null
      then (current_date - last_order_date)
    else null
  end as days_since_last_order
from public.clients;
```

**Section 8 — updated_at trigger for clients**

```sql
create or replace function public.set_updated_at()
returns trigger language plpgsql as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

create trigger clients_updated_at
  before update on public.clients
  for each row execute procedure public.set_updated_at();
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→Supabase API | All DB operations go through Supabase PostgREST, authenticated with JWT |
| JWT claims | user_metadata.role is set at user creation and read in RLS USING clauses |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | RLS clients UPDATE | mitigate | Two UPDATE policies (admin + manager lock policy) — permissive model. Phase 5 atomic locking uses WHERE locked_by IS NULL RETURNING id to prevent race conditions; field-level updates are app-enforced |
| T-02-02 | Information Disclosure | orders SELECT | mitigate | Per-role SELECT policies: manager sees only manager_id = auth.uid(); admin sees all — enforced at DB level |
| T-02-03 | Information Disclosure | call_logs SELECT | mitigate | Same pattern as orders — manager sees own rows only |
| T-02-04 | Spoofing | manager_id INSERT | mitigate | INSERT WITH CHECK (manager_id = auth.uid()) — manager cannot insert orders/call_logs attributed to another manager |
| T-02-05 | Tampering | JWT role claim | accept | Role is set by admin in Supabase dashboard; no self-registration, no way for manager to elevate own role via app |
</threat_model>
