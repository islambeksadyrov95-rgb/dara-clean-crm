## 1. БЕЗОПАСНОСТЬ

### 1.1 🔴 Утечка токена телефонии (подтверждено на проде)
**Что:** Менеджер (роль `manager`) открывает `/settings/telephony` напрямую — страница рендерит
API-токен Beeline VPBX, profileID (2978), URL вебхука и SIP-номера всех менеджеров.
**Доказательство:** залогинился тестовым менеджером на проде, страница показала токен.
**Причина:**
- `middleware.ts` — `ADMIN_ROUTES = ['/import']`, путь `/settings/telephony` не в списке → менеджера не редиректит.
- `updateTelephonySettings()` и загрузка настроек не проверяют `role === 'admin'` на сервере.
**Фикс:**
1. `middleware.ts`: `ADMIN_ROUTES = ['/import', '/settings', '/team', '/settings/telephony', '/settings/segments']`
   (проверить, что `/settings` для менеджера остаётся доступен только в части личного SIP — сейчас это так через UI-гейт, но безопаснее вынести личный SIP в отдельный route или оставить `/settings` вне ADMIN и резать секции на сервере).
2. В server actions телефонии добавить проверку роли в начало (как в `import/actions.ts:32-34`).
3. Завести util `requireAdmin()` и применить во всех admin-actions.

### 1.2 🔴 Ротация токена Beeline
**Что:** Раз токен был доступен менеджерам в проде — считаем скомпрометированным.
**Фикс:** перевыпустить X-VPBX-API-AUTH-TOKEN в ЛК Beeline, обновить env в Vercel, проверить подписку на события.

### 1.3 Что проверено и БЕЗОПАСНО (подтверждено на проде)
- `/import`, `/team` под менеджером → редирект на `/queue`. ✅
- `/settings` под менеджером → только личный SIP, admin-секции (скидки, мотивация, скрипты) не рендерятся. ✅
- RLS-изоляция: менеджер видит только свою очередь/клиентов/коммуникации. ✅
- Вебхук VPBX (`/api/vpbx/webhook`) — секрет `s` в query + 403. ✅
- Cron (`/api/cron/vpbx`) — `CRON_SECRET` Bearer + 401. ✅
- Консоль чистая, JS-ошибок нет. ✅

### 1.4 Ложные тревоги агентов (перепроверены, НЕ баги)
- «Менеджер видит все коммуникации» — НЕТ. `getCommunications()` идёт через пользовательский клиент, RLS режет. (Но явной проверки `getUser()` в action нет — защита в одну линию, желательно добавить вторую.)

## 2. КАЧЕСТВО ДАННЫХ И АРХИТЕКТУРА ИМПОРТА

> ВАЖНО: первоначальный вывод «импорт занулил суммы» был ОШИБОЧНЫМ. После чтения исходного файла
> `База Агбис.xlsx` (21389 строк, лист «Page 1») и кода парсера установлена реальная причина.

### 2.0 Что показал исходный файл (факты)
- Структура: строка 3 — заголовки `Дата приема заказа | Контрагент (крат.) | Адрес контрагента | Номер телефона | Стоимость | Услуга`.
- Дата стоит только на строках-заголовках групп («08.07.2025» + «Итого»); строки заказов наследуют дату группы.
- Всего строк-заказов: **5518**. Из них **4930 (89%) имеют `Стоимость = 0,00` в самом источнике**.
  Ненулевых — только **588** (сумма 6 768 465 ₸).
- Уникальных телефонов: **4534**. С >1 заказом: **534**. Максимум заказов на телефон: **9**.
- Пример: телефон `+77771119944` — 8 заказов 06.09→19.09→27.09→28.10→07.11→18.11→04.12→10.12 2025
  (периодичность ~раз в 2-3 недели), все «Ковёр», один адрес — но все суммы 0,00.

### 2.1 🟠 Парсер импорта КОРРЕКТЕН (не баг)
Проверен построчно (`import/page.tsx:65-169`):
- forward-fill даты из заголовков групп ✅ (стр. 80-85, 130-132)
- пропуск «Итого»/«Общий итог» ✅ (стр. 88-95)
- дедуп по телефону, агрегация orders/spent ✅ (стр. 134-153)
- `Math.round` для денег ✅ (стр. 163)
- парс «0,00» → 0 ✅ (стр. 109)
Импорт корректно поймал 588 ненулевых строк → 614 клиентов с `total_spent>0` в CRM. Сходится.

### 2.2 🟠 Источник без сумм (бизнес-проблема, не код)
89% строк выгрузки Агбиса не содержат стоимость. Без сумм LTV, средний чек, финансовый эффект скидок
останутся нулями независимо от кода.
**Фикс:** запросить в Агбисе отчёт с заполненной колонкой «Стоимость» (другой отчёт/право доступа/период),
затем переимпорт.

### 2.3 🟠 ГЛАВНОЕ: импорт схлопывает историю в агрегаты
`import/actions.ts` пишет в `clients` только агрегаты: `total_orders, total_spent, avg_order_value,
last_order_date` + ОДИН адрес. Поштучные заказы НЕ сохраняются как строки.
**Последствия (то, что нужно бизнесу, но недоступно):**
1. Поштучная история теряется → периодичность («заказывает раз в 3 недели») не посчитать.
2. Колонка **«Услуга» вообще не импортируется** — в парсере нет `iService` (только дата/имя/телефон/адрес/сумма).
   Что заказывал клиент (Ковёр/Мебель/Шторы) — потеряно.
3. Множественные адреса схлопываются в один (первый непустой).
4. Суммы по заказам (даже 588 ненулевых) хранятся только суммарно.
В карточке клиента «Заказы (0)», т.к. таблица `orders` содержит только заказы, созданные менеджерами в CRM
(на проде — 1 строка), а импортированная история туда не пишется.
**Фикс:** см. Фаза 1 ниже (таблица истории заказов).

## 3. НЕДОДЕЛКИ (фичи есть, но недостижимы/мертвы)

### 3.1 🟠 Карточка клиента недостижима из списка (исходный баг)
**Что:** `/clients/[id]` исправна (открыл по прямому URL — рендерит шапку, метрики, заказы, звонки),
но в `/clients` имя клиента не ссылка; клик по строке и кнопка «Выбрать» открывают боковую панель звонка,
а не карточку. Достижима только из `/orders` (клик по строке) и тоста входящего звонка.
**Фикс (по user flow, не удалять):**
- Имя клиента в таблице `/clients` → ссылка на `/clients/[id]`.
- В боковой панели звонка → ссылка «Открыть карточку».
- Клик по строке оставить как есть (панель звонка), чтобы не сломать рабочий флоу.

### 3.2 🟡 Мёртвые страницы заказа/WhatsApp → встроить в flow
**Что:** `/queue/order/[clientId]` и `/queue/whatsapp/[clientId]` — на них ноль ссылок.
Это легаси (коммит 578b2d4), заменённое инлайн-панелями `OrderForm`/`WhatsAppPanel` в `/queue`.
Сейчас это дубль логики скидок (та же формула в двух файлах).
**Фикс (решение пользователя — не удалять, а вписать):**
- На карточке клиента добавить кнопки «Оформить заказ» → `/queue/order/[clientId]` и «WhatsApp» →
  `/queue/whatsapp/[clientId]`. Сейчас с карточки нельзя ни заказать, ни написать — только позвонить.
- Сами страницы переписать как тонкие обёртки над общими компонентами `OrderForm`/`WhatsAppPanel`
  (загрузка клиента + рендер общей формы). Дубль логики скидок исчезает.
- После заказа со страницы — возврат на карточку; из очереди, как сейчас, — «Следующий клиент».

### 3.3 🟠 Глобальный поиск — мёртвый
**Что:** «Поиск клиента, заказа…» в шапке — это `StaticText`, не input. Клик ничего не делает.
**Фикс:** сделать командную палитру (Ctrl+K): поиск по телефону/имени → карточка/панель из любого места.

### 3.4 🟠 Скрипты звонков не доходят до менеджера
**Что:** руководитель в настройках пишет скрипты по 5 сегментам с переменными `{имя}/{дней}/{скидка}`,
но в панели звонка `/queue` они не показываются нигде.
**Фикс:** в панели звонка показывать скрипт сегмента активного клиента с подстановкой переменных.

### 3.5 🟢 Мёртвый код
`getInboxWhatsAppLogs()` в `inbox/actions.ts:46` — не вызывается ниоткуда. Удалить.

### 3.6 🟠 Полумиграция ролей
**Что:** миграция `20260611000004_role_to_app_metadata.sql` перенесла роль в `app_metadata` (БД/RLS),
но 17 файлов кода читают `user_metadata.role`. Только `settings/segments` (самая новая) читает `app_metadata`.
Миграция **не закоммичена** (untracked в git) — возможно, не применена/не задеплоена.

## 4. ЛОГИКА / МЕТРИКИ

### 4.1 🟠 «Отправить WhatsApp» = звонок
**Что:** нажатие «Отправить WhatsApp» (из «Дозвонился» и «Не дозвонился») пишет диспозицию,
которая увеличивает счётчик «Звонки» (проверено на проде: 0→1 без реального звонка).
**Последствие:** искажает конверсию (звонки → заказы) в KPI менеджера и мотивации.
**Фикс:** отделить WhatsApp-действие от счётчика звонков, либо считать звонком только реальный звонок
(disposition reached/not_reached, не sent_whatsapp).

### 4.2 🟡 Воронка считает неверно
**Что:** на `/pipeline` «База 1000 → В очереди 1000 (100% от базы)». «1000» — потолок выборки, не вся база (4879).
«В очереди» фактически считает «клиенты с заказом в периоде» (`pipeline/page.tsx:54-69`), а не очередь.
Сквозная конверсия 0% при наличии заказа. Цифры некликабельны.
**Фикс:** считать по полной базе (без скрытого лимита), исправить определение «В очереди»,
сделать этапы кликабельными (→ список клиентов этапа), добавить разрез по менеджерам.

## 7. СТРУКТУРА (best-practice CRM, целевая модель)

Принципы зрелых CRM повторных продаж (amoCRM/Kommo, Pipedrive, HubSpot), наложенные на Dara Clean:

1. **Клиент — центр, всё кликабельно.** Любое упоминание клиента ведёт в карточку. → сейчас 🔴 (3.1).
2. **Карточка = рабочее место (хаб + действия + единый timeline).** Из карточки — звонок/WhatsApp/заказ/задача;
   timeline объединяет звонки + заказы + рассылки + WhatsApp одной лентой. → сейчас 🔴 (нет действий, лента раздельная).
3. **Следующее действие у каждого активного клиента** (`next_action_at` + заметка). Очередь = задачи, а не база.
   Закрывает «пропустить → вернуться позже». → сейчас 🟡 (только callback).
4. **Авто-логирование коммуникаций** (система пишет факты, менеджер — смысл). Звонки через VPBX — да;
   WhatsApp вслепую (wa.me), статусы Wazzup не подтягиваются. → сейчас 🟡.
5. **Воронка = сделки, сегменты = клиенты — не путать.** Сегмент (кому звонить) есть и хорош;
   сущности «сделка» нет — для короткого цикла обзвона это осознанное MVP-упрощение.
6. **Кабинет руководителя — 4 вопроса на одном экране:** что сегодня / идём ли по плану (+прогноз) /
   где теряем (воронка по менеджерам, причины отказов) / качество (AI-оценки звонков).
   Плюс ведомость фактически начисленных бонусов. → сейчас 🔴 (нет дашборда, нет прогноза, нет ведомости).
7. **Роли — 4 слоя защиты + один источник чтения роли.** → сейчас 🟡 (см. 1.1, 3.6).
8. **Гигиена данных:** телефон — натуральный ключ (E.164 есть), история неизменяема, импорт с превью+откатом.

### Сильные стороны (на уровне рынка или выше)
Обязательная диспозиция звонка; лок от дублей (осталось показать); AI-генерация WhatsApp (работает на проде);
AI-оценка звонков 1-10; RFM-сегментация с настраиваемыми правилами; мотивационный калькулятор с грейдами и джекпотом.

## 9. ФАЗА 1 — ИСТОРИЯ ЗАКАЗОВ (детальный план)

### Цель
Клиент в карточке показывает полную историю: каждый заказ (дата, сумма, услуга, адрес), периодичность,
итоги. Сегменты считаются точнее. Импорт перестаёт терять данные.

### 9.1 Почему отдельная таблица, а не `orders`
Таблица `orders` (id, client_id, manager_id, services[], amount, discount_percent, discount_amount, comment, created_at):
- `manager_id` обязателен — у исторических заказов менеджера нет;
- `created_at` = момент создания записи, а нужна ИСТОРИЧЕСКАЯ дата заказа;
- нет адреса на уровне заказа (адрес мог меняться);
- смешивать импорт с боевым потоком заказов (state-машина, мотивация, воронка) — риск исказить KPI.
Решение: отдельная таблица `order_history` для импортированной истории. Боевые заказы остаются в `orders`.

### 9.2 Схема таблицы (миграция)
> РЕШЕНИЕ по деньгам: переходим на `integer` (ЦЕЛЫЕ ТЕНГЕ, без тиын) — правило database.md (money = integer).
> Все денежные колонки мигрируют numeric(12,2) → integer (см. Фаза «Фундамент» §11). `order_history.amount`
> создаётся сразу как `integer`.
> РЕШЕНИЕ по типам: заводим `gen:types` + `types/database.ts` (supabase.md type-chain) — см. §11. После этого
> источник правды по типам — database.ts; до этого — SQL-схема.

```sql
CREATE TABLE order_history (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  client_id uuid NOT NULL REFERENCES clients(id) ON DELETE CASCADE,
  order_date date NOT NULL,                          -- историческая дата (из группы Агбиса)
  amount integer NOT NULL DEFAULT 0 CHECK (amount >= 0),  -- ЦЕЛЫЕ ТЕНГЕ, 0 если нет в источнике
  service text,                                      -- «Ковёр»/«Мебель»/… из колонки Услуга
  address text,                                      -- адрес на момент заказа
  source text NOT NULL DEFAULT 'agbis_import' CHECK (source IN ('agbis_import','manual')),
  import_batch_id uuid,                              -- для отката конкретного импорта
  created_at timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_order_history_client_id ON order_history(client_id);
CREATE INDEX idx_order_history_client_date ON order_history(client_id, order_date DESC);
CREATE INDEX idx_order_history_batch ON order_history(import_batch_id);

ALTER TABLE order_history ENABLE ROW LEVEL SECURITY;
-- SELECT: менеджер видит историю своих клиентов, админ — всю
CREATE POLICY oh_select ON order_history FOR SELECT TO authenticated
  USING (
    (auth.jwt() -> 'app_metadata' ->> 'role') = 'admin'
    OR EXISTS (SELECT 1 FROM clients c WHERE c.id = order_history.client_id
               AND c.assigned_manager_id = auth.uid())
  );
-- INSERT/DELETE: только admin (импорт идёт под admin через service role; политика — для прямых вызовов)
CREATE POLICY oh_write ON order_history FOR ALL TO authenticated
  USING ((auth.jwt() -> 'app_metadata' ->> 'role') = 'admin')
  WITH CHECK ((auth.jwt() -> 'app_metadata' ->> 'role') = 'admin');
```
Денежная единица: `integer` (целые тенге). Math.round до целого при любом расчёте.
