n# Сервис записи, транскрипции и диаризации звонков — архитектура и план

> Рабочий документ для реализации. Составлен многоагентным проектированием + адверсариальным ревью (2026-06-18). Ревью переупорядочило приоритеты: сначала ценность на текущем моно, тяжёлая инфра — опционально и последней.

## 0. Как читать этот документ

Два главных вывода ревью, на которых стоит всё остальное:

1. **Балл за технику продаж не зависит от диаризации** (скоринг читает плоский текст). Значит самое ценное и дешёвое — не запись по каналам, а **сценарный слой + язык-гард** на уже работающем моно-пути. Прямо сейчас система скорит автоответчики, недозвоны и галлюцинации Whisper как «плохую работу менеджера» — это реальный вред баллу уже на проде.
2. **Нативный двухканальный рекордер — золочение.** Он даёт только точные UI-метки спикеров и метрику «доля речи», которых нет в целях сервиса. Это недели работы + вечная поддержка 5 десктопов. Поэтому он отложен как явное бизнес-решение владельца, а не первый шаг.

Документ ведёт от немедленной ценности (Фаза 1, чистый код) к опциональному «правильному пути» (рекордер, Фаза 4 под вопросом).

## ★ ФИНАЛ — решения владельца и план реализации (ГЛАВНЫЙ раздел; заменяет §8, уточняет §6/§7/§10)

### Решения (зафиксировано 2026-06-18, владелец)
1. **Согласие на запись НЕ требуется**; передача голоса/транскрипта в **Groq (США) принята**. Юр-блокеры сняты. (Дёшево и без UI: включить в аккаунте Groq no-training/zero-retention, если доступно.)
2. **Кросс-источниковый дедуп — делаем.**
3. **«Облако без аудио».** Аудио в Supabase постоянно НЕ храним. Постоянно в облаке — только транскрипт + метаданные (мало). Аудио-мастер живёт у источника: MicroSIP — на ПК менеджера, Beeline — на сервере Beeline. Облачная копия транзитная (на время транскрипции + короткое окно для недавнего прослушивания) → потом удаляется. Старое прослушивание — re-fetch из источника по уникальной ссылке записи. Это разом решает: рост Storage, годовой signed URL, право на удаление, orphan-by-delete.
4. **Чанкинг НЕ нужен, модель не меняем.** Факт: 8кГц-моно ~26 kbps → лимит Groq 25МБ = **~135 мин** (100МБ dev = ~9ч) — недостижим. Реальный риск только таймаут обработки длинного файла → поднять таймаут + воркер. Split ломает хранение/прослушивание — не делаем. (Self-hosted Whisper — опц. fallback, не сейчас.)
5. **Все high+medium чиним сразу** (ничего не откладываем, кроме нативного рекордера и self-hosted Whisper).

### Модель «облако без аудио» (следствие п.3)
- В строке звонка постоянно: `transcript`, `transcript_segments`, `language`, `score`, `scenario_tags`. Плюс `audio_ref` (как заново достать: MicroSIP `{recorder_uid, filename}`, Beeline `{parentUUID}`) + `audio_cache_path` (nullable, транзитная копия) + `audio_cache_until`.
- Пайплайн: получить аудио (по `audio_ref`) → транскрибировать → сохранить транскрипт → пометить кэш коротким TTL (или удалить сразу).
- Прослушивание: кэш жив → короткий signed URL; иначе re-fetch (Beeline `GetCallRecordContent`; MicroSIP — запрос демону переслать файл с ПК) → временный URL. **Честный предел MicroSIP:** re-fetch старого требует ПК менеджера онлайн с подключённой папкой — поэтому кэш-окно (напр. 30 дн.) покрывает «недавнее», старое — по запросу.
- Долгоживущие signed URL не пишем вообще (старый годовой фикс субсумирован этой моделью).

### План по фазам (все high+medium внутри, под решения)

**Фаза 1 — Фундамент, целостность, «облако без аудио»**
- Единый `lib/recordings/pipeline.ts` `processRecording`; `actions.ts`/`vpbx/transcribe.ts` → тонкие врапперы (убрать дубль).
- Модель `audio_ref`/кэш вместо durable signed URL (субсумирует старую Фазу 0).
- **Кросс-источниковый дедуп** перед транскрипцией: связать `vpbx_calls`↔`call_logs` по `(менеджер/телефон, started_at ±60с)`, скорить физический звонок ОДИН раз на строке, где есть аудио. Контекст: Beeline-запись сейчас ВЫКЛ (аудио идёт от MicroSIP) → сейчас не двоит, но гард обязателен на случай включения Beeline-записи и для нестыдвоения статистики. Победитель при двух аудио — подтвердить у владельца (дефолт VPBX).
- **Атомарный claim**: `UPDATE ... transcription_status='processing' WHERE status IN ('pending','failed') RETURNING id` (+'processing' в CHECK); усилить слабый MicroSIP-гард.
- **Раздельный персист**: транскрипт пишем сразу после Whisper ('transcribed'); скоринг — отдельная стадия (провал скоринга не теряет уже полученный транскрипт).
- **Таймаут**: Whisper AbortSignal 60с→180с; `export const maxDuration=300`; тяжёлую работу через `after()` на serverless не оставлять.
- **Язык-гард kk** (порог = доля казах-слов, не наличие букв); **TZ Asia/Almaty** (`ALMATY_OFFSET` из `lib/agbis/order-dates.ts`); метка `audio_quality='narrowband'`; `pipeline_version`/`processed_at`.

**Фаза 2 — Сценарный слой + справедливость**
- `lib/transcription/scenarios.ts`: детекторы 0-7; **IVR = совокупность (маркер И нет клиентских сегментов)**, не ИЛИ; осторожные пороги too_short.
- Колонки `scenario_tags`, `score_skip_reason`, `transcript_segments`; UI-диалог Менеджер/Клиент + ярлыки классов.
- **Ручной оверрайд**: `reclassifyCall(id, forceScorable)` + `score_override_by/at` + кнопка admin «это разговор — оценить» (идемпотентность пропускает `manual_override`).
- **Прозрачность**: персистить strengths/improvements + показать менеджеру + «Не согласен» → `score_disputed`.
- Правило агрегата: средний балл при N≥10 оценённых; рядом доля null; `call_score` НЕ в премию до валидации детекторов.

**Фаза 3 — OPS на Railway (Vercel-кроны 2/2 заняты)**
- **Cleanup транзитного аудио** (TTL кэша) + **re-fetch playback** (Beeline GetCallRecordContent; MicroSIP daemon re-upload).
- **Recovery-reconcile** (failed/застрявшие/attached-not-transcribed) + **429 backoff + семафор** concurrency 1-2.
- `recording_access_log` (аудит доступа при выдаче URL); вьюха `daily_detector_breakdown` + алерт >50% non-conversation/день; **labeling-loop** (`verdict_correct` + `getDetectorAccuracy`); **feature-flags** `TRANSCRIPTION_ENABLED`/`SCORING_ENABLED`; health-счётчики.

**Фаза 4 — Опционально (решение владельца):** нативный dual-рекордер; self-hosted Whisper как no-limit fallback.

## 1. Цель и принцип

### Цель сервиса
Записывать звонки менеджер↔клиент, транскрибировать (RU/KK/смешанный), по возможности разделять кто говорит, корректно обрабатывать пограничные сценарии (чтобы не-разговоры не портили статистику) и оценивать технику продаж (балл 1–10 + summary + извлечение «откуда узнали»).

### Принцип 1 — балл устойчив к ошибкам диаризации (честно)
`scoreCall` получает **плоский текст** (`corrected`). Перемешанные метки «менеджер/клиент» балл не ломают: LLM определяет роли по смыслу (кто предлагает услугу/называет цену = менеджер). Метки спикеров нужны только для UI-диалога и метрик, а не для балла. Это страховка: даже кривая эвристика спикеров не искажает оценку.

### Принцип 2 — канальное разделение = «правильный путь», но не обязательный
Идеальная диаризация = раздельные каналы (менеджер = микрофон near-end, клиент = выход гарнитуры far-end), физически раздельные на ПК через WASAPI. Это даёт 100% точные метки спикеров без эвристики. Но это улучшает только UI-метки и talk-ratio, которых нет в целях. Поэтому канальное разделение — опциональная цель Фазы 4, включается только если по факту работы на моно руководителю не хватит именно точных меток/talk-ratio.

### Принцип 3 — split на сервере не нужен (свойство, а не повод для рекордера)
- **Моно (работает СЕГОДНЯ):** вход — уже один смешанный трек (Beeline `audio/mpeg`, MicroSIP моно-MP3). Резать нечего. ffmpeg не нужен.
- **Dual (будущее, опционально):** рекордер отдаёт два готовых моно-файла. Резать каналы не нужно — транскрибируем два файла отдельно и сливаем по таймкодам.

serverless-чистота — свойство текущей архитектуры, а не аргумент за рекордер.

## 2. Архитектура целиком

### Компоненты
- **Источники звонков (есть):** Beeline VPBX (webhook CallStart/CallFinish + запись смешанного файла) и MicroSIP на ПК (моно-MP3 в папку `Record call/` + браузерный синк через File System Access API в Storage).
- **Единый пайплайн (новое, ядро Фазы 1):** `lib/recordings/pipeline.ts` — одна функция `processRecording`, через которую идут ОБА источника. Сейчас логика скачивания/транскрипции/скоринга скопирована один-в-один в `actions.ts` и `vpbx/transcribe.ts` (реальный дубль). Пайплайн его устраняет.
- **Язык-гард (новое):** kk-фоллбэк в транскрипции.
- **Сценарный слой (новое):** `lib/transcription/scenarios.ts` — чистые функции-детекторы, решают «скорить / не скорить / какие метки».
- **Скоринг (усиление):** `scoreCall` получает метки сценариев как контекст-подсказки, не как основу балла.
- **Хранение (минимальное):** новые колонки в существующих `call_logs` и `vpbx_calls` (НЕ новая таблица — см. §6.1).
- **Рекордер (опционально, Фаза 4):** нативный Windows-рекордер, пишущий два моно-файла в ту же папку `Record call/`, которые подхватывает существующий синк.

### Поток данных (моно-путь — то, что реально работает)

```
[Звонок состоялся]
   ├── Beeline VPBX: webhook CallFinish (isRecorded, recordUrl, номер, направление, времена)
   │        → vpbx_calls (строка звонка уже скоррелирована с клиентом/менеджером)
   └── MicroSIP: моно-MP3 в "Record call/" на ПК менеджера
            → браузерный синк (sync-client.ts) грузит в Storage local/<uid>/
            → attachLocalRecording матчит к call_logs (клиент по телефону из имени / по времени)
                          │
                          ▼
        ┌─────────────────────────────────────────────┐
        │  lib/recordings/pipeline.ts                 │
        │  processRecording(source, callRow)          │
        └─────────────────────────────────────────────┘
                          │
        1. Скачать аудио (signed URL по запросу, НЕ годовой)
                          │
        2. transcribeAudio (Whisper large-v3)
             ├─ автодетект языка
             ├─ ЯЗЫК-ГАРД: если язык ∉ {ru, kk} → 1 доп. вызов force kk,
             │             выбор ru/kk простым правилом (казах-буквы)
             ├─ applyDictionaryFixes (словарь ~440 правил)
             └─ assignSpeakers(segments, startSpeaker)  ← якорь по направлению
                          │
        3. classifyCall(segments, corrected, direction, durationSec, finishStatus)
             ├─ Детектор 1: no_speech (недозвон/тишина)
             ├─ Детектор 2: hold_music / whisper_hallucination → чистка сегментов
             ├─ Детектор 3: ivr / voicemail (по правилам-маркерам)
             ├─ Детектор 4: too_short
             └─ метки: overlap / code_switch / multi_manager / language
                          │
              ┌───────────┴────────────┐
              ▼                        ▼
   verdict.scorable = false    verdict.scorable = true
   → score = null,             → scoreCall(cleanedTranscript, scenarioTags)
     score_skip_reason            → score 1-10 + summary + acquisition
              └───────────┬────────────┘
                          ▼
        4. Запись результата в call_logs / vpbx_calls:
           transcript, transcript_segments (jsonb!), language,
           scenario_tags, score, summary, score_skip_reason
```

Ключевая правка: `transcript_segments` сейчас вычисляются, но НЕ сохраняются — в БД пишется только плоский `transcript`. Это единственный реальный пробел в данных, который надо закрыть.

## 3. Компонент — Сценарный слой (Фаза 1, наибольшая ценность)

Делаем первым: отсутствие этих проверок прямо сейчас портит средний балл менеджера ложными данными.

### Где стоит
Чистая функция между транскрипцией и скорингом, внутри `pipeline.ts`. НЕ ходит в сеть для 90% звонков (арифметика + регэкспы). LLM-классификатор IVR — не реализуем в v1 (правил достаточно; добавить позже только если данные покажут пропуски).

### Контракт
```typescript
// lib/transcription/scenarios.ts
export type CallClass = 'conversation' | 'non_conversation' | 'no_speech' | 'too_short'
export type CallFlag =
  | 'overlap' | 'hold_music' | 'whisper_hallucination'
  | 'code_switch' | 'multi_manager' | 'voicemail_prompt' | 'short_call'

export type ScenarioVerdict = {
  callClass: CallClass
  scorable: boolean                 // === 'conversation'
  flags: CallFlag[]
  languageMix: 'ru' | 'kk' | 'mixed'
  directionAnchor: 'manager_first' | 'client_first'
  cleanedSegments: ChatSegment[]    // после чистки галлюцинаций/холд-музыки
  cleanedTranscript: string         // → идёт в scoreCall (без мусора)
  skipReason: string | null
}
```

### Детекторы и пороги

- **Детектор 0 — нормализация направления (всегда).** `inbound → client_first`, `outbound/internal → manager_first`. Передаётся в `assignSpeakers` стартовым спикером. Сейчас всегда стартует с `manager` — для входящих это ошибка меток.
- **Детектор 1 — `no_speech` (skip, score=null).** `finishStatus ∈ {NOT_ANSWERED, BUSY, CANCELLED}` ИЛИ пустой `corrected` ИЛИ суммарная речь `< 1.0с` ИЛИ значимых слов `< 3`.
- **Детектор 2 — `hold_music` / `whisper_hallucination` (чистка сегментов).** Удаляем мусор ДО скоринга: повтор фразы `≥3` подряд или `≥4` всего; сегмент `>30с` при `<20` слов или скорость `<0.5` слов/с; зацикленная n-грамма `>50%` слов; словарь известных галлюцинаций Whisper («Продолжение следует…», «Субтитры сделал…», «Спасибо за просмотр», «Подписывайтесь на канал»); длинный сегмент (`>15с`) вне RU/KK = музыка. После чистки `<3` слов → каскад в `no_speech`.
- **Детектор 3 — `non_conversation` (IVR/автоответчик, score=null).** Регэксп-маркеры RU/KK («оставьте сообщение после сигнала», «абонент недоступен», «нажмите 1», «абонент жауап бермейді», «байланыс аймағынан тыс») + структурный признак (один спикер + outbound + 5–25с).
- **Детектор 4 — `too_short` (не скорим).** `durationSec < 20` И чередований `< 3` И значимых слов `< 25`. Лучше честный `null`, чем выдуманная «4/10».
- **Детектор 5 — язык (метка).** `mixed` если RU и KK оба `≥15%`, иначе доминирующий. Балл от языка не зависит.
- **Детектор 6 — overlap (метка «не штрафовать»).** Моно-прокси: соседние сегменты разных спикеров с зазором `<0.3с`, доля плотных стыков `>10%`. В промпт: «есть перебивания, не снижай балл за обрывочность».
- **Детектор 7 — multi_manager (метка).** Маркеры перевода («передаю коллеге», «соединяю со специалистом»). Звонок скорим, но в промпт: «несколько сотрудников, оценивай компанию в целом».

### Сводка: класс → действие
| callClass | scorable | scoreCall | score в БД | UI-ярлык |
|---|---|---|---|---|
| `conversation` | да | вызывается с `cleanedTranscript` | 1–10 | балл + summary |
| `non_conversation` | нет | пропуск | null | «Автоответчик / IVR» |
| `no_speech` | нет | пропуск | null | «Не дозвонились» |
| `too_short` | нет | пропуск | null | «Короткий звонок» |

### Константы (named)
```typescript
const SCENARIO = {
  MIN_SPEECH_SEC: 1.0, MIN_SIGNIFICANT_WORDS: 3,
  SHORT_CALL_MAX_SEC: 20, SHORT_CALL_MAX_TURNS: 3, SHORT_CALL_MAX_WORDS: 25,
  HALLUCINATION_REPEAT_CONSECUTIVE: 3, HALLUCINATION_REPEAT_TOTAL: 4,
  LONG_SEGMENT_SEC: 30, LONG_SEGMENT_MAX_WORDS: 20, SLOW_SPEECH_WORDS_PER_SEC: 0.5,
  OVERLAP_GAP_SEC: 0.3, OVERLAP_RATIO_THRESHOLD: 0.10,
  CODE_SWITCH_MIN_RATIO: 0.15, IVR_OUTBOUND_MAX_SEC: 25,
} as const
```

## 4. Компонент — Пайплайн транскрипции + язык-гард

### 4.1. Единый пайплайн (рефактор — первый шаг Фазы 1)
Сделать `lib/recordings/pipeline.ts` `processRecording` ПЕРВЫМ, встроить язык-гард и сценарный слой внутрь него ОДИН раз. Тогда оба источника (MicroSIP, VPBX) получают улучшения бесплатно, и устраняется текущий дубль (`loadClientContext` + обвязка `scoreCall` + acquisition-стор скопированы идентично в `actions.ts` и `transcribe.ts`). Эти файлы становятся тонкими врапперами.

### 4.2. Язык-гард — kk-фоллбэк (упрощённый)
```typescript
const ALLOWED = new Set(['russian', 'kazakh'])

async function transcribeWithLanguageGuard(audio, fname) {
  const auto = await whisperRequest(audio, fname)            // автодетект
  if (!auto.text) return { ...auto, language: 'unknown' }
  if (ALLOWED.has(auto.language))
    return { ...auto, language: auto.language === 'kazakh' ? 'kk' : 'ru' }

  // язык ∉ {ru, kk} → один доп. вызов force kk (ru уже косвенно отвергнут)
  const kk = await whisperRequest(audio, fname, 'kk')
  // простое правило: больше казах-букв ә/ғ/қ/ң/ө/ұ/ү/һ/і → kk, иначе ru
  return hasMoreKazakhLetters(kk.text) ? { ...kk, language: 'kk' } : forceRu(...)
}
```
Убирает лишний Whisper-вызов на канал (было до 3, стало до 2) и две словарные функции. Текущий код уже делает force-ru одной строкой — меняем целевой язык фоллбэка с ru на kk.

### 4.3. Таймаут / ретраи / идемпотентность
- Whisper: `AbortSignal.timeout(60_000)` + `withRetry` (2 ретрая, backoff 1с→2с + джиттер, только на сетевых/429/5xx; на 400 — не ретраить). Скоринг: 20с. Выставить `export const maxDuration`.
- Идемпотентность на уровне строки БД: непустой `transcript` / `transcription_status='done'` → skip.

### 4.4. Структура выхода (обратносовместима)
`transcribeAudio` продолжает отдавать `{ raw, corrected, segments }`. Добавляем сохранение `segments` (jsonb) + `language`. `merged_dialog` для UI собирается из `segments` на чтении.

## 5. Скоринг

Контракт не меняется: вход — плоский `cleanedTranscript` (без галлюцинаций холд-музыки). Метки сценариев — контекст-подсказки.

Дополнения к промпту `scoreCall`:
- «Разметка спикеров приблизительна — определяй роли по смыслу.»
- При `overlap`/`hold_music` — не считать искажённые фрагменты словами-паразитами; при `transfer`/`multi_manager` — оценивать основного менеджера; при обрыве — не снижать балл за отсутствие «закрытия сделки».
- При `ivr/voicemail/no_speech/too_short` — `scoreCall` вообще не вызывается (экономия токенов, чистота метрик).
- `acquisition_answer` («откуда узнали») извлекается как сейчас.

## 6. Данные, матчинг, безопасность, плеер

### 6.1. Решение по данным — НЕ создавать `call_recordings` сейчас
Не строим единую таблицу `call_recordings` и не затеваем Expand/Contract миграцию двух живых таблиц. Это enterprise-паттерн не по масштабу 5 менеджеров, оправдан ТОЛЬКО для dual-рекордера (два storage-пути на звонок). Раз рекордер отложен — откладывается и таблица.

Минимум — аддитивные колонки в ОБЕ существующие таблицы (`call_logs`, `vpbx_calls`):
```sql
-- одна аддитивная миграция; gen:types в том же коммите (One Window Rule)
ALTER TABLE call_logs  ADD COLUMN transcript_segments jsonb;
ALTER TABLE call_logs  ADD COLUMN language text CHECK (language IS NULL OR language IN ('russian','kazakh','mixed'));
ALTER TABLE call_logs  ADD COLUMN scenario_tags text[] NOT NULL DEFAULT '{}';
ALTER TABLE call_logs  ADD COLUMN score_skip_reason text;
-- аналогично vpbx_calls (score/summary там уже есть)
```
Закрывает реальный пробел (потеря спикер-разметки) с нулевым риском и без переписывания чтения. Единая `call_recordings` — позже и только под dual-рекордер.

### 6.2. Матчинг (моно-путь)
- **VPBX:** источник истины — события (`number_a/b`, `started_at`, `vpbx_uuid`). `client_id/manager_id` уже скоррелированы. Доп. матчинг не нужен.
- **MicroSIP:** клиент по телефону из имени файла (`findClientIdByFilename`), иначе по `manager + время ±окно`.
- **Усиление:** при матчинге по времени для звонков подряд (< интервала окна) требовать совпадения телефона клиента, а не только близости по времени — иначе запись привяжется к чужому клиенту (текущий `ORDER BY created_at DESC LIMIT 1` молча берёт ближайший-но-неверный).
- **Осиротевшие записи:** запись без матча сейчас теряется (`call_logs.client_id NOT NULL`). Мера в Фазе 1: файл остаётся в Storage, помечается обработанным, OPS-счётчик «осиротевших» это видит.

### 6.3. Безопасность

**ФАЗА 0 (сразу, независимо) — годовой signed URL.** `call_logs.audio_url` хранит signed URL с TTL **1 год** (`SIGNED_URL_TTL_SECONDS = 60*60*24*365` в `actions.ts:15`) — доступ к записи (PII: ФИО + телефоны) в обход RLS на год. Фикс:
1. перестать писать долгоживущие URL — хранить `storage_path`;
2. генерировать короткий signed URL (15–60 мин) по запросу с проверкой владельца через RLS;
3. занулить/проэкспайрить существующие `audio_url` в проде.

**RLS на любых новых полях/таблицах — app_metadata, НЕ user_metadata.** `user_metadata` перезаписывается самим пользователем (`supabase.auth.updateUser({data:{role:'admin'}})` → самоповышение до admin). Любая ролевая проверка читает `(auth.jwt() -> 'app_metadata' ->> 'role')`. Зеркалить ИСПРАВЛЕННЫЕ политики `vpbx_calls` из `20260611000004`, не оригинал. Новые колонки в существующих таблицах наследуют их (уже корректную) RLS.

**Прочее:** транскрипт = PII (RLS таблицы); в логи — только `id`, не текст; токены в env.

### 6.4. Плеер
- **Моно:** один файл, играет в оба уха. Готово. Signed URL по запросу.
- **Dual (Фаза 4):** запрещено отдавать manager.wav в одно ухо — микс-вниз на клиенте через Web Audio API (два `MediaElementSource` → один `GainNode` → моно в оба канала), синхрон по общему `currentTime`.

## 7. Эксплуатация и стоимость

### 7.1. OPS — лимит cron исчерпан
`vercel.json` уже декларирует ДВА крона (`/api/cron/vpbx`, `/api/cron/agbis-orders`). Vercel Hobby = 2 крона максимум — проект уже на лимите. Оба ночные — не ловят «рекордер мёртв в рабочее время».
- Никаких новых vercel-кронов. Мониторинг записей вынести на **Railway-воркер** (там Telegram-бот, нет лимита, можно intra-day).
- Если что-то должно жить на Vercel — складывать в существующий `/api/cron/vpbx`.

### 7.2. Наблюдаемость
Server action `getRecordingsHealth()` (только админ, app_metadata-роль) + страница в настройках: счётчики за 24ч/7д; застрявшие `transcribing`/`failed` (`attempts<3` → ретрай, `>=3` → dead-letter); осиротевшие файлы; структурные логи `[recordings] {event, id, source, status}` без текста.

### 7.3. Стоимость Groq
~5 менеджеров × ~40 звонков/день × 22 дня ≈ ~4 400/мес, после отсева ~3 000 скоримых, средний разговор ~3 мин.
- **Whisper large-v3** (~$0.04/час): 220 ч → **~$9/мес** (моно). Язык-гард kk-фоллбэк = +1 вызов только на чуже-детектнутых (редко) → копейки.
- **llama-3.3-70b скоринг** (~$0.0012/звонок × 3 000) ≈ **~$4/мес**.
- **Итого моно: ~$13/мес.** Dual удваивает аудио-минуты (+$9) ИЛИ транскрибировать только смешанный для текста, а каналы для таймкодов (без удвоения).

## 10. Пробелы, найденные при проверке полноты (и куда их вписать)

> Проверка против кода (`lib/vpbx/transcribe.ts`, `lib/recordings/*`, `app/(protected)/pipeline/actions.ts`, `vercel.json`, миграции) подтвердила перечисленные пробелы на реальных строках, а не теоретически. Топ-5, что нельзя пропустить: (1) согласие на запись + трансграничная передача Groq (блокеры легальности); (2) кросс-источниковый дедуп (иначе скоринг задвоится в день 1); (3) retention + каскад в Storage (закон + бесконечный рост); (4) maxDuration + Railway-воркер (иначе длинные звонки и `after()` тихо теряются); (5) ручной оверрайд + измеримость детекторов (иначе система обнуляет людей без права на пересмотр).

### Фаза 0 (безопасность / предусловия) — БЛОКЕРЫ запуска Фазы 1
- **Правовое основание записи.** Пре-роллинг «разговор записывается» на стороне Beeline АТС (без кода) + обязательная фраза менеджера на исходящих MicroSIP. Предусловие Фазы 1. (Закон РК №94-V ст.7-8)
- **Трансграничная передача в Groq (США).** Зафиксировать в docs zero-retention/no-training DPA Groq, покрывающий ОБА потока — `audio/transcriptions` И `chat/completions`. Согласие на передачу за рубеж — в текст уведомления. Self-hosted Whisper — строкой в roadmap как fallback (НЕ реализовывать сейчас). (ст.16)
- **Согласие сотрудника на ИИ-оценку.** Положение/допсоглашение о контроле качества, подписанное менеджерами до включения скоринга. Score — инструмент развития, не единственное основание кадровых решений.

### Фаза 1 (базовый pipeline)
- **maxDuration на путях транскрипции.** `export const maxDuration = 300`. Тяжёлую работу через `after()` на serverless НЕ оставлять (режется по wall-clock — транскрипция тихо умирает после 200).
- **Атомарный claim вместо read-then-write.** `UPDATE ... SET transcription_status='processing' WHERE ... AND status IN ('pending','failed') RETURNING id`. Добавить 'processing' в CHECK. MicroSIP-путь (гард только `if (transcript)`, actions.ts:118) усилить.
- **Раздельный персист транскрипта и скоринга.** После Whisper сразу писать transcript+language+segments статусом 'transcribed'; скоринг — отдельная стадия. Провал скоринга НЕ затирает транскрипт (сейчас затирает — transcribe.ts:124-126).
- **Таймзона периода Asia/Almaty.** Переиспользовать `ALMATY_OFFSET='+05:00'` из `lib/agbis/order-dates.ts`. Сейчас pipeline/actions.ts:42 считает в UTC → вечерние звонки Алматы уезжают «на завтра».
- **Метка качества `audio_quality='narrowband'`** для 8кГц-моно; не штрафовать в промпте. (Денойз/апсэмплинг — НЕ делать.)
- **pipeline_version smallint + processed_at** в обе таблицы — предпосылка для управляемой переобработки.

### Фаза 2 (детекторы / справедливость / дедуп)
- **Кросс-источниковый дедуп одного звонка.** Перед транскрипцией MicroSIP-записи проверять `vpbx_calls` того же менеджера/клиента в окне ±60с с is_recorded; скорить ТОЛЬКО один источник. «Какой источник победитель» — БИЗНЕС-РЕШЕНИЕ владельца. (Иначе 2 балла за 1 звонок — массово в день 1.)
- **Дедуп MicroSIP в БД, не в localStorage.** Признак «обработано» по (manager_id, storage_path) в БД; `attachLocalRecording` upsert-идемпотентным по storage_path.
- **Ручной оверрайд зарубленных звонков.** `reclassifyCall(callId, forceScorable)` + `score_override_by/at` + кнопка «Это был разговор — оценить» (admin). Идемпотентность пропускает `manual_override=true`.
- **Прозрачность оценки.** Персистить strengths/improvements (уже считаются); показывать менеджеру балл+summary+зоны роста+запись; кнопка «Не согласен» → `score_disputed`.
- **Детектор 3 (IVR) — совокупность, не ИЛИ.** IVR-маркер И нет клиентских сегментов. Подозрительные → флаг needs_review, но всё равно scoreCall.
- **Порог языка kk как ДОЛЯ слов** с казах-буквами (≥N%), не наличие отдельных букв (иначе русский звонок с «Жанна»/«Қонаев» → ложный kk → каскад в no_speech).
- **Длинные звонки — чанкинг НЕ нужен (решение владельца + факт).** При 8кГц-моно ~26 kbps лимит Groq 25МБ = ~135 мин (100МБ dev = ~9ч) — недостижим для звонков химчистки. Split ломает хранение/прослушивание → не делаем. Фикс только: поднять таймаут Whisper (60с→180с) + воркер + scenario `too_long` (>~120 мин) как страховка вместо падения в failed.

### Фаза 3 (OPS на Railway-воркере — Vercel-кроны исчерпаны 2/2)
- **Retention аудио + cleanup.** `RECORDING_RETENTION_DAYS` (90). Раз в сутки удалять objects старше N дней + занулять audio_url/storage_path. (Нарушение supabase.md §Append-Only.)
- **Каскад удаления клиента чистит Storage.** В `deleteClient`: собрать storage_path → storage.remove() → удалить клиента. (DB-каскад до файла не доходит — у storage.objects нет FK.)
- **Право на удаление по запросу.** `deleteCallDataByClient(client_id)` — файл из Storage + зануление transcript/score/summary.
- **Recovery-reconcile.** failed/застрявшие/attached-but-not-transcribed старше X мин → ре-триггер с backoff и attempts++ → dead_letter + алерт.
- **Backoff на Groq 429** (Retry-After, 3-5 попыток+jitter) + семафор concurrency 1-2 на воркере.
- **Аудит доступа к записям.** `recording_access_log(actor_id, call_log_id, client_id, accessed_at)` при генерации signed URL.
- **Наблюдаемость детекторов.** Вьюха `daily_detector_breakdown` + строка в health + алерт при доле non-conversation > 50%/день.
- **Измеримость детекторов (labeling-loop).** Колонка `verdict_correct` + 2 кнопки у admin + `getDetectorAccuracy()`. 20-30 меток/нед.
- **Feature-flags.** `TRANSCRIPTION_ENABLED` и `SCORING_ENABLED` (раздельно). Мгновенный откат без редеплоя.
- **Storage в TCO.** Реальный TCO ≠ $13/мес: ~1.5 ГБ/мес рост, Supabase Free (1 ГБ) исчерпается за ~3 нед → **Pro обязателен**; с retention 90 дн. установившийся ~5 ГБ.

### Фаза 4 (валидация / тесты / отчётность)
- **Интеграционный тест на реальных сэмплах.** Gated `it.skipIf(!GROQ_API_KEY)`: 3-4 фикстуры из `test-audio/` через РЕАЛЬНЫЙ pipeline; ассертить инварианты (язык∈{ru,kk}, too_short→score=null), не точный текст. `ru-chekhov-31min` — заодно проверка чанкинга.
- **Правило агрегата score=null.** Средний балл только при N≥10 оценённых/период; рядом доля null. ЯВНО: НЕ заводить call_score в премию до накопления baseline и валидации детекторов.
- **Стратегия бэкфилла (в DECISIONS).** force-reprocess по pipeline_version или пометить старые pipeline_version=0.
