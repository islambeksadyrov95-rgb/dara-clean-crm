## Формат addons в SaveOrderForAll (ПОДТВЕРЖДЁН вживую)
Услуга: `{ dos_id, tovar_id:"100387", count:"<площадь м²>", addons:[...] }`. Поле addon:
```
{ addon_id: "100241", values: "1002336" }   // строковый id выбранной опции для vt=8
{ addon_id: "100242", values: "2|2|3" }      // фигура-строка для vt=9
```
- Ключ поля = `addon_id` (НЕ addon_type_id). Значение = `values` (строка).
- Эти варианты дали err1: `addon_type_id` («addon_id не существует»),
  `str_value_id`/`value_str`/`value` («Не задано значение доп реквизита values!»).
- `values:"1002336"` (id опции) — принято без ошибки.

## БЛОКЕР: цена не применяется
- При корректных addons заказ создаётся (dor 100284), но `Srvices[0].price=0`, `qty=6`, `debet=0`.
- Явное поле `price` в услуге ИГНОРИРУЕТСЯ (price=0 и при `price:"1500"`, и при `"9000"` — dor 100285/100286).
- Т.е. ни тип (change_price=1 → value_flt), ни явная цена не дают строке ненулевую цену через `SaveOrderForAll`.

### Проверено 2026-06-16 (всё дало price=0)
- `SaveOrderForAll` с addons (тип Миллионник 1002337 + фигура `"2|3|2|"`) → price 0, qty 6.
- Явная цена в услуге под ключами `price`/`serv_price`/`price_serv` → ИГНОРИРУЕТСЯ.
- `UpdateOrderForAll` (повторное сохранение для recalc) → price 0.
- Реальный ценовой ковёр (dor 1033716, 09.06): price=1500 (=value_flt типа), qty=6. Цель ясна,
  но raw-API её НЕ выставляет на создании.
- **Рабочая гипотеза (НЕ проверена):** Агбис проставляет цену ковра при обработке цехом
  (смена статуса/recalc в приложении), а не на создании. Тогда: CRM шлёт тип+площадь+addons,
  показывает менеджеру ОЦЕНКУ `value_flt(тип) × площадь`, а авторитетную цену подтянет read-sync
  (зеркало `agbis_*`). Проверить: создать ковёр, дождаться обработки, посмотреть price в read-back.

### Гипотезы для следующей попытки (не проверены)
1. Цена для editable-позиций ставится отдельной командой ПОСЛЕ создания (`UpdateOrderForAll`
   с ценой услуги, или команда пересчёта/recalc по dor_id).
2. `values` для vt=8 должен быть иной структурой, чтобы Агбис применил `value_flt` (массив объектов,
   ссылка на value_flt, и т.п.).
3. У ковров отдельный прайс-лист (не price_id=0), где цена/м² уже зашита на tov_id.
4. **Спросить у владельца:** как ставится цена ковра в самом Агбисе (через тип? вручную приёмщиком?
   пересчёт?). Это бизнес-знание — money-critical, требует валидации (rules: financial logic).

### Что готово к реализации, как только цена решится
- CRM МОЖЕТ показать цену в форме сам: `площадь × value_flt(тип)` (данные уже в AddonTypes).
- UI: выбор типа ковра (10 опций с ценой), выбор фигуры + размеры → площадь, превью суммы.
- Пуш: добавить ветку в `buildOrderItems`/`push-order` с addons `{addon_id, values}` + механизм цены (п.1–3).
