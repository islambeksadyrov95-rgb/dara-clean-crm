## 0. Корректировки: дока Агбиса врёт (проверено)

| Что | Дока Агбиса говорит | ЖИВОЙ API отдаёт | Где |
|-----|---------------------|------------------|-----|
| `tp` выезда в `Trips`/`Trip`/`order_plans` | `0 забрать / 1 доставить` | **`1 забрать / 2 доставить`** (нулей нет ни разу в 302 выездах) — тот же enum, что у `TripOrder` | 04-double-session.md:14 |
| `mp_status` в `Trips` | `0/1/2` | **полный набор `0–6`** (видели 0,1,2,3,6 живьём) | 04-double-session.md:16,125 |

## 1. Жизненный цикл ЗАКАЗА (`status_id`, проверено)

```
1 Новый → 3 В исполнении → 4 Исполненный → 5 Выданный
                                    ⮑ 7 Отменённый (терминальный)
```
- Реальные значения в выборке: `1,3,4,5,7` (2 «На хранении» и 6 «Закрытый» в окне не встретились).
- `date_out_fact` (факт. выдача, `dd.mm.yyyy HH:MM:SS`) проставляется при переходе в **5 Выданный**.
- `current_sclad_id` = физическое местоположение: машина (`1023`/`1032`) → `1` (цех «Сайрам») → машина.
- Распределение за 30 дней: Новый 74 / В исполнении 103 / Исполненный 52 / **Выданный 637** / Отменённый 71.
- У всех боевых заказов: `kind_id=0` (химчистка), `price_list_id=0`, `firm_id=11`.

## 2. Жизненный цикл ВЫЕЗДА (`mp_status`, отдельная сущность)

```
0 Новый → 3 Принял → 4 В пути → 5 Приехал → 1 Совершён
                                  ⮑ 2 Отменён    ⮑ 6 Пересогласование
```
- `tp`: **1 = забрать** (грязные у клиента, в день приёма) · **2 = доставить** (чистые клиенту, в день выдачи).
- `kind_id` выезда: 1 по заказу · 2 по клиенту · 3 по складу.
- Выезд и заказ живут независимо: бывает отменённый выезд при активном заказе и наоборот.
- На заказ — до двух выездов (забор + доставка); самовывоз = выезда нет (склад приёма = точка).

## 4. Следствия для read-back (двусторонний синк)

- **Статус заказа** (`orders`/`order_history`): инкрементальный `OrderByDateTimeForAll` по окну `update`, апдейт по `agbis_order_id`/`agbis_dor_id`. `LastChangeOrder` не работает.
- **Статус выезда** (`order_trips`): `Trip {id=agbis_trip_id}` для линка к заказу + `Trips detail:1` по дню для `mp_status` (его нет в `Trip`). Маппинг `tp`→`kind`: **1→pickup, 2→delivery** (как на записи, инвертировать НЕ надо). `mp_status=2` → выезд отменён.
- `order_plans` использовать нельзя для финального статуса выезда (скрывает совершённые/отменённые).

## 5. Сценарии: отмена / перенос / комментарии (проверено на 853 выездах)

Распределение `mp_status` за 17 дней: совершён 678 · **отменён 113** · новый 50 · принял 6 · **пересогласование 6**.
`comment` заполнен у **277/853 (32%)** выездов — это рабочий канал данных, его НАДО читать и показывать.

### 5.1 Отмена ЗАКАЗА (status 7)
- У отменённых заказов `FullOrderInfo` отдаёт **пустые** `order_comments`, `order_services`, `order_plans`.
  Причина отмены НЕ в заказе — она в комментарии **выезда**. Отмена заказа каскадит отмену его выездов
  (видели выезды с `mp_status=2` при `dor_status=7`).

### 5.2 Отмена ВЫЕЗДА (mp_status=2) — структурированный комментарий
Агбис при отмене дописывает в `comment` выезда строку фиксированного формата:
```
<Имя сотрудника> отменил выезд: "<причина>"
ID выезда на пересогласование: <new_trip_id>
```
Реальные причины (примеры): «Ковер не готов», «Не готов», «На завтра», «Перенос», «30.06 перенос»,
«Клиент сама позвонит», «Возврат вещей в цех», «Клиент не ответил». Язык — RU + KK вперемешку.

### 5.3 Перенос (Пересогласование) — это НЕ редактирование даты, а cancel+recreate ⚠
Перенос выезда в Агбисе = **отменить старый выезд (mp_status=2) + создать НОВЫЙ на новую дату**, связанные
комментариями-ссылками:
- старый выезд: `… ID выезда на пересогласование: <new_id>`
- новый выезд: `Выезд создан на основе отменённого: <old_id>` (часто в статусе `mp_status=6 пересогласование`)
Один заказ может переноситься многократно → цепочка выездов. **Доказано на цепочке (dor 1062062, доставка):**
`1038858(15.06,отменён)→100252(16.06,отменён)→100258(17.06,отменён)→100269(18.06,отменён)→100274(20.06,НОВЫЙ)`
— TripID каждый раз НОВЫЙ, дата у существующего выезда не меняется. Один заказ переносился до **8 раз**.
Обнаружено 68 цепочек среди 853 выездов.

> **Правило read-back (доказано):** актуальный выезд плеча = **последний НЕ отменённый** (`mp_status≠2`) выезд
> цепочки для (dor_id, tp). Все `mp_status=2` — история переносов. На `order_trips` пишем ОДНУ строку = актуальный.

**Причины отмен несут бизнес-события:** «Перестирка»/«2-ковер перестирка» (**перемойка** — отдельный сценарий!),
«Отмена заказа», «Ковер не готов», «калада емес» (KK «нет в городе»). Забор переносится так же, как доставка.
Инструкции («ЖК Метрополь, заезд с ул.Попова») протягиваются в новые выезды цепочки.

> **Следствие для интеграции (важно):** наш `syncArm`/`editArmTrip` сейчас правит выезд in-place
> (`TripOrder` id+mp_status=0). Это расходится с тем, как реально работает персонал (cancel→recreate
> со ссылочными комментами). Для паритета: (а) read-back должен схлопывать цепочку cancel→new в ОДИН
> актуальный выезд на `order_trips` (брать последний не-отменённый); (б) при отмене из CRM писать
> причину в `comment`; (в) при переносе из CRM — либо cancel+recreate как Агбис, либо хотя бы
> корректно отображать чужие переносы.

### 5.4 Комментарии выезда — рабочие данные (читать и показывать)
Кроме отмен, `comment` несёт инструкции водителю/клиенту: время («Забрать после 18 часов», «ЗАБРАТЬ ДО ОБЕДА»),
доступ («вход с южной стороны, левый серый звонок», «ЖК Аль-Фараби», «Частный дом»),
оплату/нюансы («27720 тг надо взять», «Мокрые ковры, коэффициент 1,5»), контакты доп.лиц.
Для паритета: синкать `comment` выезда в `order_trips` и показывать в карточке заказа.

### 5.5 Прочие невычитанные сценарии заказа (для полноты паритета)
- `waiting_confirm` (0/1/2/3) — подтверждение клиентом (видели 0 и 2).
- `is_not_confirmed` (0/1) — подтверждение сотрудником.
- `condition_id`/`condition_name` — состояние изделия (с 4.4.0).
- Услуги заказа имеют свой `status_id` каждая (частичная готовность).
- Промежуточные статусы выезда водителя: 3 принял · 4 в пути · 5 приехал.
- Оплаты: `doc_type` (касса/карта/банк/бонус/депозит), долг `dolg`.

## 7. Перестирка / перемойка (проверено — 9 заказов за ~30 дней)

Повторная химчистка ковра. **Нет отдельного статуса или поля** — фиксируется ТОЛЬКО в `comment` выезда
(«Перестирка», «2-ковер перестирка»). Два паттерна на реальных данных:

- **A. Перестирка на ЗАБОРЕ.** Выезд `tp=1 забрать` с комментом «Перестирка» — повторный забор уже
  готового/выданного ковра на перемойку (00485-6, 00807-6, 00626-6: забор «Перестирка» → заказ снова в цеху).
- **B. Перестирка на ДОСТАВКЕ.** Выезд `tp=2 доставить` отменяется с причиной «Перестирка» → «Возврат вещей в цех»
  → ковёр обратно в цех, потом новый выезд доставки (00569-6: `1038644`«Возврат в цех»→`100200`«Перестирка»→…).

**Эффект на данные:**
- Статус заказа/услуги **откатывается НАЗАД** (видели Исполненный→«В исполнении»; заказы снова `status=3`).
- `current_sclad_id → 1` (Сайрам, цех) — ковёр физически вернулся в работу.
- Создаётся новый выезд доставки (cancel+recreate, §5.3).
- **Частичная перестирка:** на заказе с неск. коврами один может быть «5 Выдан», другой снова «4/3» (00569-6: услуги `["4","5"]`).

> ⚠ **Следствие для state-machine (важно!):** переходы статуса заказа **НЕ линейны** — заказ может идти

> при read-back **НЕ должна отклонять «обратные» переходы**. Перестирку детектить по комменту выезда
