# Промт: повторный прогон 2 тест-задач выездного обслуживания на проде (Agbis)

> Готовый промт для сессии/агента. Цель — на боевом Agbis создать тестовые заказы с привязанными
> выездами по реальным сценариям, проверить отображение, затем убрать за собой. Все грабли вшиты.

## КОНТЕКСТ (доказано, не переоткрывать)
- Привязать выезд (`MOBILE_PLAN`) к заказу (`DOCS_ORDER`) через публичный REST API **нельзя** (у `TripOrder`
  нет параметра заказа; проверено исчерпывающе). Связь = строка в junction-таблице
  `MOBILE_PLAN_ORDERS(ID, MOBILE_PLAN_ID, DOR_ID, DEP_ID, LAST_DEP_ID, DEP_SRC_ID)` в **локальной Firebird**,
  которая реплицируется на центральный сервер (~5 мин) и видна в REST + десктопе.
- Этот офис = **депо 3** (Dara). Узел филиальный, репликация на триггерах: любой INSERT в
  `MOBILE_PLAN_ORDERS` ловится в очередь `MST_META_CHANGES` и выгружается `AgentRepl` в `Repl/Out7to1`.
- Десктоп читает ЛОКАЛЬНУЮ базу → привязка видна в нём СРАЗУ после локальной вставки.

## ДОСТУПЫ
**Firebird (локально, read concurrent-safe):** 64-бит клиент `C:\fb64client\fbclient.dll` (НЕ бандловый 32-бит).
Пароль в `C:\Agbis\LicensingService.ini` `[Firebird] Password=`.
```python
import re, pathlib
from firebird.driver import connect, driver_config
pwd = re.search(r"Password=(.+)", pathlib.Path(r"C:\Agbis\LicensingService.ini").read_text(errors="ignore")).group(1).strip()
driver_config.fb_client_library.value = r"C:\fb64client\fbclient.dll"
con = connect("127.0.0.1/3050:C:/Agbis/DB/ARM_7.FDB", user="SYSDBA", password=pwd)
```
**REST (creds в `.env.local`: AGBIS_API_BASE/USER/PWD):** Login → Session_id. Read-команды бесплатны.
GET-параметры: `?Cmd=`+`urllib.parse.quote(json.dumps(params))`+`&SessionID=`. POST-тело:
`{Cmd: body, SessionID}`. Ответы URL-encoded — декодировать (`urllib.parse.unquote(s.replace('+',' '))`).

## РЕЦЕПТ ПРИВЯЗКИ (рабочий, с КРИТИЧНЫМ нюансом ID)
plain SYSDBA (current_role=NONE). FK: `MOBILE_PLAN_ID→MOBILE_PLAN.ID`, `DOR_ID→DOCS_ORDER.ID` —
ОБЕ строки должны быть в ЛОКАЛЬНОЙ базе (дождаться долива через репликацию).
```sql
INSERT INTO MOBILE_PLAN_ORDERS (ID, MOBILE_PLAN_ID, DOR_ID, DEP_ID, LAST_DEP_ID, DEP_SRC_ID)
VALUES (:atomic_id, :trip_id, :order_id, 3, 3, 3);
```
**ID — САМОЕ ВАЖНОЕ (иначе коллизия с центром, привязка не приживётся):**
- Префикс ID обязан = `103` (депо 3). НЕ давать BI-триггеру авто-ID (он берёт `GEN_CUR_DEP_ID=107`).
- **НЕ использовать `MAX(ID)+1`** — выбранное значение может быть локальным «пробелом», но ЗАНЯТЫМ на
  центре → конфликт (проверено: так привязка осталась на фантоме `1062187`).
- Брать ID **выше глобального high-water mark** в депо-3 диапазоне `1030000–1039999`, заведомо
  никем не использованный (напр. `select max(ID) from MOBILE_PLAN_ORDERS where DEP_SRC_ID=3` и
  взять с запасом +50, проверив отсутствие в `MOBILE_PLAN_ORDERS` И в `MST_META_CHANGES`). В идеале —
  атомарно через генератор Агбиса. Суффикс держать ≤ 9999 (чтобы префикс оставался `103`).
- При перепривязке: СНАЧАЛА вставить новый junction, ПОТОМ удалить старый (иначе `CDO`-триггер
  при удалении последнего junction поставит выезду `mp_status=2` = отмена).

## ГРАБЛИ (учтены)
- **Кириллица в API:** через JS `JSON.stringify` (боевой код) сырой UTF-8 сохраняется ПРАВИЛЬНО.
  Через Python `json.dumps` по умолчанию `ensure_ascii=True` → `\uXXXX` → Агбис сохранит ПУСТО!
  В Python ВСЕГДА `json.dumps(..., ensure_ascii=False).encode('utf-8')` ЛИБО url-кодировать значения
  `urllib.parse.quote(value)`. Касается имён клиентов (`ContragForAll`) и `Comments`.
- **Телефон:** `ContragForAll` дедупит по телефону; `+7705618170` Агбис нормализует в `+77705618170`
  (добавляет 7). Сверять формат с существующим клиентом, иначе создашь дубликат. Перед созданием
  искать существующего: `ContragInfoForAll{teleph_cell}` + `ClientsByDateTimeForAll` по имени.
- **order_plans:** `FullOrderInfo.order_plans` показывает только ЖИВЫЕ выезды (`mp_status` 0/3/4);
  «Совершён»(1)/«Отменён»(2) скрыты. Привязку верифицировать через `Trip{id}.orders[]` (надёжный канал).
- **Репликация:** вход (центр→локаль) и выход (локаль→центр) по ~5 мин. Полный цикл REST-создания →
  привязки → подтверждения в REST ~10-12 мин. Десктоп — сразу после локальной вставки.
- **SaveOrderForAll:** `Comments` — массив url-encoded строк или `[]` (иначе reject). `Services` —
  `[{tovar_id, count:'1', addons:[]}]`, tovar_id брать из `PriceList{price_id:'0'}` где `tovar_type=='2'`.
  Склады: машины `1032`(М1)/`1023`(М2), цех `1`. doc_num/doc_date — в таблице `DOCS` (join по `DOC_ID`).

## ЗАДАЧА 1 — сценарии выездного обслуживания
1. Прочитать заказы за 15 дней в локальной Firebird (`DOCS_ORDER ↔ MOBILE_PLAN_ORDERS ↔ MOBILE_PLAN`),
   вывести таксономию сценариев с частотами. Канонический набор (должен покрываться):
   **забор+доставка** (≈73%), **доставка только**, **забор только**, **самовывоз** (без выезда),
   **перенос** (цепочка: отменён→пересогласование→новый), **перестирка** (коммент «Перестирка»),
   **отменённый заказ** (status 7), **забор+2 доставки**.
2. По каждому сценарию: REST `SaveOrderForAll` (клиент 10042 «ТЕСТ Сценарий удалить» или новый тест-клиент)
   → REST `TripOrder` нужные выезды (tp=1 забор / tp=2 доставка, нужные `mp_status`) → дождаться долива
   заказа+выездов в локаль → вписать junction(ы) по РЕЦЕПТУ (атомарный ID!) → верифицировать.
3. Верификация: локально junction'ы DEP 3,3,3 + в `MST_META_CHANGES`; REST `Trip.orders[]` = нужный заказ;
   десктоп (открыть заказ → «Выезды текущего заказа»). Состав выездов = сценарию (самовывоз=0, перенос=3…).

## ЗАДАЧА 2 — клиенты (существующий + новый) с заказом и выездом
1. **Существующий клиент** (напр. Исламбек): СНАЧАЛА найти его в базе по телефону/имени (`ContragInfoForAll`,
   `ClientsByDateTimeForAll`). Реальный Исламбек = `contr_id 1041860`, тел `+77057618170`. Использовать
   ЕГО `contr_id`, НЕ создавать дубликат. Создать заказ на него + АКТИВНЫЙ выезд (доставка, `mp_status=0`,
   чтобы попадал и в `order_plans`). Привязать. Проверить: имя клиента, телефон, услуга, склад, дата,
   выезд — всё отображается в десктопе и в API.
2. **Новый клиент:** создать `ContragForAll` с именем (правильная кодировка кириллицы!) + телефоном →
   заказ + активный выезд → привязка → та же полная проверка отображения.
